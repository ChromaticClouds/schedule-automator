# Codex Requirements: Schedule Automator MVP

## 1. Product Goal

Build a personal AI productivity scheduler.

The app helps the user convert weekly goals into daily executable schedules.
It uses Google Calendar as the source of existing fixed events and writes approved AI-generated events into a dedicated Google Calendar named `AI Schedule`.

AI must only generate schedule drafts.
The user must approve a draft before it is synced to Google Calendar.

## 3. Core Product Decisions

* Personal-use MVP.
* Mobile-first app.
* Google Calendar integration is required.
* AI uses Gemini API.
* AI draft generation happens on the server only.
* Gemini API key must never be exposed to the mobile app.
* User input style is natural language + command syntax.
* Weekly goal planning comes first.
* Daily schedule draft is generated every day at `wakeTime + 10 minutes`.
* Daily total work time limit is 8 hours.
* Missed tasks are rescheduled into the remaining time of the current week.
* User marks missed tasks during end-of-day review.
* AI-generated calendar event titles must use a prefix such as `[AI]`.

## 4. MVP User Flow

### 4.1 Weekly Planning

User enters weekly goals.

Examples:

```txt
My goals for this week are to improve the portfolio README, refactor the inventory, and look into five job postings.
```

```txt
/goal Improve this week's portfolio README
/task Inventory refactoring 2 hours, Importance 5.
```

The server sends structured context to Gemini.
Gemini returns task breakdowns.
The server validates the response before saving.

### 4.2 Daily Draft Generation

At `wakeTime + 10 minutes`, the server generates a daily schedule draft.

Required input context:

* user settings
* weekly goals
* tasks
* missed tasks
* protected times
* existing Google Calendar events
* current week remaining availability

Output:

* schedule draft
* schedule blocks
* warnings
* assumptions
* unresolved items

The draft is stored in MongoDB with status `draft`.

### 4.3 Draft Approval

User can:

* approve
* edit
* regenerate
* reject

Only approved drafts are synced to Google Calendar.

The server must prevent duplicate calendar events by storing `calendarEventId`.

### 4.4 End-of-Day Review

User marks tasks as:

* done
* missed

Missed tasks:

* increase `postponedCount`
* return to rescheduling queue
* are placed into remaining weekly availability
* become `overflow` if not schedulable this week

## 5. Functional Requirements

### Goals

* User can create weekly goals.
* Goal has title, description, importance, status, week range.
* Goal status: `active`, `paused`, `completed`, `archived`.
* AI can break goals into executable tasks.

### Tasks

* User can create tasks manually or via AI breakdown.
* Task can belong to a goal.
* Task has estimated minutes.
* Task can have deadline.
* Task has importance, goalImpact, postponedCount.
* Task status: `todo`, `scheduled`, `done`, `missed`, `overflow`, `archived`.
* Missed task increments postponedCount.

### Protected Time

Protected time blocks prevent unrealistic schedules.

Examples:

* sleep
* meal
* shower
* leisure
* exercise
* custom

Protection levels:

* `hard`: cannot be touched
* `soft`: can be adjusted only with user approval

Sleep defaults to `hard`.

### Schedule Draft

* Schedule draft is generated by AI but validated by server logic.
* Draft status: `draft`, `approved`, `synced`, `rejected`, `expired`.
* Daily total scheduled work must not exceed 480 minutes.
* Schedule blocks must not overlap.
* Schedule blocks must not collide with Google Calendar events.
* Schedule blocks must not collide with hard protected time.
* Existing approved AI calendar events must be treated as occupied.

### Calendar

* User can connect Google account.
* Server reads existing Google Calendar events.
* Server creates or finds dedicated `AI Schedule` calendar.
* Approved schedule blocks are written to `AI Schedule`.
* Generated event titles must include prefix from env:

```env
GOOGLE_CALENDAR_EVENT_PREFIX=[AI]
```

* Store `calendarEventId` per synced block.
* Re-approving the same draft must not create duplicate events.

### Chat Input

Support natural language and commands.

The mobile app uses a chat-style composer as the primary planning input. It must:

* keep the current conversation scrollable above a bottom-anchored composer
* avoid the keyboard, home indicator, and app chrome with Safe Area handling
* expose planning, review, explore, and settings categories in a left drawer
* support multiline input and a 44dp-or-larger send action with an accessible label
* show pending, success, clarification, and error responses as distinct message states
* preserve the unsent draft while a request fails or the user navigates away
* work at 360px, 390px, and 430px phone widths without clipped controls

Commands:

```txt
/goal <goal text>
/task <task text>
/protect <time rule>
/plan tomorrow
/replan this week
/review today
```

Server parses chat input into structured command payload.
If parsing fails, return a clarification prompt.

For daily schedule creation, the composer may send an optional 1-500-character
`instruction` with `POST /schedule-drafts`. The server treats it only as a
scheduling preference: it must not alter the AI extraction schema, task ID
allowlist, conflict checks, or daily work limit. The mobile client preserves
the text until a successful response replaces the empty-state composer.

## 6. AI Requirements

AI provider: Gemini API.

The mobile app must never call Gemini directly.

Server responsibilities:

* build Gemini request payload
* validate request payload
* call Gemini
* validate Gemini response
* reject malformed output
* repair only safe and deterministic errors
* never trust AI output directly

AI output must be structured JSON. Daily schedule generation uses a deliberately
small extraction schema so Gemini only returns fields that the model must decide.
The server hydrates all trusted display and persistence fields from known context.

Required daily schedule extraction response:

```ts
type AiScheduleExtractionResponse = {
  blocks: {
    taskId: string;
    start: string;
    end: string;
  }[];
};
```

The server must map `taskId` to the known task title and set the block type. It
must generate stored summary, assumptions, and warnings from validated server
results; Gemini must not be asked to return those fields. Goal breakdown and chat
command parsing use separate small schemas and must not be combined with daily
schedule extraction.

Validation rules:

* `start < end`
* no overlapping blocks
* no protected time collision
* no existing calendar event collision
* taskId must exist in the supplied schedule context
* total work time <= 480 minutes
* block duration must be reasonable
* invalid AI output must not be saved as approved

## 7. Scheduling Priority

Use this priority formula unless replaced later:

```txt
priorityScore =
deadlineUrgency * 0.35
+ importance * 0.25
+ goalImpact * 0.25
+ postponedCount * 0.15
```

Clamp postponed score to avoid old low-value tasks dominating the schedule.

Suggested:

```txt
postponedScore = min(postponedCount, 5)
```

## 11. Security Rules

* Never put secrets in mobile env.
* Never expose Gemini API key to the client.
* Never expose Google client secret to the client.
* Encrypt Google OAuth tokens before saving.
* Treat Google Calendar event text as untrusted input.
* Treat AI output as untrusted input.
* Validate all AI input and output with Zod.
* Do not store raw AI payload logs by default.
* Do not commit `.env`, `.env.local`, credentials, tokens, keystores, or generated native folders.

## 12. MVP Out of Scope

Do not implement these in MVP unless explicitly requested:

* team scheduling
* multi-user collaboration
* Gmail event extraction
* Notion integration
* Todoist integration
* location-based travel time
* voice input
* advanced analytics dashboard
* automatic calendar writes without user approval
* production push notification system
